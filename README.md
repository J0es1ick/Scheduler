# Scheduler

Telegram-бот автоматически загружает актуальные снимки расписаний:

- ИГХТУ — <https://www.isuct.ru/student/schedule>;
- ИГЭУ — <http://schedule.ispu.ru/>.

Данные сохраняются в PostgreSQL, после чего бот выдаёт расписание выбранной
группы. Для ИГЭУ номер вводится в формате `1-40`, для ИГХТУ — `3/147`.

В проект также входит отдельная веб-админка Scheduler Control Room. Она показывает
состояние источников, историю проходов, группы, занятия и пользователей, позволяет
менять интервал и безопасно запускать синхронизацию вручную.

## Запуск

1. Заполните `.env` по образцу `.env.example`.
   Для запуска Go-процессов на хосте оставьте `DATABASE_HOST=127.0.0.1` и
   укажите опубликованный порт PostgreSQL. В Docker Compose адрес базы
   автоматически заменяется на `postgres:5432`.
2. Запустите PostgreSQL:

   ```powershell
   docker-compose up -d postgres
   ```

3. При необходимости выполните первый импорт отдельно:

   ```powershell
   go run ./cmd/sync
   ```

   Команда без аргументов обновляет оба вуза. Один источник можно запустить
   отдельно: `go run ./cmd/sync ispu-main` или `go run ./cmd/sync isuct-main`.

4. Запустите бота:

   ```powershell
   go run ./cmd/bot
   ```

5. В другом терминале запустите админку:

   ```powershell
   go run ./cmd/admin
   ```

   Откройте `http://localhost:8080` и используйте `ADMIN_ACCESS_TOKEN` из `.env`.
   Этот ключ нужен как начальный и аварийный способ входа; не публикуйте его и
   задайте длинным случайным значением.

Миграции применяются автоматически. Фоновый worker запускает проверку сразу при
старте, затем каждые пять минут проверяет, пришло ли время обновить источник.
ИГХТУ и ИГЭУ обновляются раз в час; неуспешный проход повторяется на следующей
проверке. Полный первичный импорт занимает несколько минут.

## Подписки и уведомления

Профиль пользователя хранится в PostgreSQL и не сбрасывается после перезапуска
бота или контейнера. `/start` восстанавливает основную группу автоматически.

- выбор новой группы добавляет подписку, не удаляя прежние;
- `/settings` и `/subscriptions` показывают все подписки, позволяют назначить
  основную группу, удалить подписку и включить или выключить уведомления;
- команды `/today`, `/tomorrow`, `/week` и `/twoweeks` используют основную группу;
- при реальном изменении итогового расписания бот уведомляет всех подписчиков
  группы. Простая повторная загрузка тех же данных уведомление не создаёт;
- изменения из редактора также создают уведомления после успешной фиксации в БД;
- доставки хранятся в PostgreSQL, повторяются после временных ошибок Telegram и
  не теряются при перезапуске. После отключения уведомлений или удаления подписки
  ожидающие сообщения отменяются.

Количество «подписок» в админке — это число групп, изменения которых отслеживает
пользователь. Это не число входов, сообщений или Telegram-каналов.

## Редактор расписания

Раздел «Редактор» показывает итоговое расписание выбранной группы по дням недели. Администратор может
добавлять занятия, менять данные существующих записей, скрывать ошибочные занятия и восстанавливать
версию источника. Ручные изменения хранятся отдельно от снимка парсера, поэтому не теряются после
следующего автоматического обновления. Бот читает итоговое представление расписания и видит изменения
сразу после сохранения.

Группа выбирается поиском по всему справочнику, а не коротким списком вариантов. Если расписание
содержит занятия по чётным и нечётным неделям, режим «Все» выводит две отдельные недельные доски;
общие занятия присутствуют в обеих. Благодаря этому пары с одинаковым временем не выглядят как
несколько одновременных занятий одной недели. Закрытые перечисления — вуз, тип занятия, подгруппа,
статус и вид недели — остаются обычными селекторами. Произвольный интервал обновления источника
вводится числом от 5 до 10 080 минут.

## Горячая линия

Команда `/hotline` и кнопка «Горячая линия» в главном меню позволяют пользователю выбрать один из
двух сценариев: сообщить об изменении уже подключённого расписания или предложить новое учебное
заведение. Бот выдаёт подходящий шаблон и сохраняет заполненное обращение в PostgreSQL.

- одновременно у пользователя может быть не более трёх открытых обращений;
- все действующие администраторы получают уведомление, а бывшие администраторы исключаются из
  доставки непосредственно перед отправкой;
- заявки рассматриваются в разделе «Обращения» админки: их можно принять в работу или отклонить;
- результат и комментарий администратора сохраняются и доставляются пользователю через надёжную
  очередь с повторными попытками после временных ошибок Telegram;
- решение фиксируется в административном журнале.

## Структура веб-интерфейса

Исходный код React находится в корневом каталоге `web/admin`: страницы — в `src/pages`, общие
компоненты — в `src/components`, хуки — в `src/hooks`, слой API — в `src/api`. Каталог
`internal/adminui/dist` содержит только результат production-сборки и маленький Go-адаптер `embed.go`.
Он расположен внутри `internal`, потому что `go:embed` может встраивать только файлы из дерева своего
пакета. Переносить туда исходники React не нужно, а выносить собранные файлы за пределы Go-пакета
смысла нет.

## Админка в Telegram

Назначение роли `is_admin` само по себе не подключает Mini App — оно только
разрешает пользователю пройти серверную проверку при запуске из Telegram.

Для подключения:

1. Опубликуйте админку по доступному из интернета HTTPS-адресу.
2. Запишите этот адрес в `ADMIN_PUBLIC_URL` и перезапустите сервисы.
3. Откройте диалог с ботом и отправьте `/start` или `/admin`. Бот автоматически назначит администраторам
   персональную кнопку меню с адресом `ADMIN_PUBLIC_URL`; если Telegram временно недоступен, настройка
   будет повторяться каждые 30 секунд.

Сервер проверяет подпись `initData`, срок её действия и флаг `users.is_admin`;
пользователь с ролью администратора входит без `ADMIN_ACCESS_TOKEN`.

Первого Telegram-администратора можно назначить напрямую в PostgreSQL:

```sql
UPDATE users SET is_admin = TRUE WHERE id = '<telegram_id>';
```

Последующие роли меняются в разделе «Пользователи». Все административные
изменения пишутся в отдельный журнал аудита. Ручной запуск и фоновый worker
защищены PostgreSQL advisory lock, поэтому один источник не парсится одновременно
из двух процессов.

Для полного запуска через Docker:

```powershell
docker-compose up -d --build
```

Интерфейс собирается из `web/admin`, а готовые файлы встраиваются в Go-бинарник,
поэтому отдельный Node.js-сервер в продакшене не нужен.

## Проверка

```powershell
go test ./...
```

Live-тест реального сайта (полный список групп и расписание контрольной группы):

```powershell
$env:ISUCT_INTEGRATION_TEST='1'
go test ./internal/scrapper/isuct -run TestLiveISUCTAdapter -v -count=1
```

Живой тест ИГЭУ:

```powershell
$env:ISPU_INTEGRATION_TEST='1'
go test ./internal/scrapper/ispu -run TestLiveISPUAdapter -v -count=1
```

Сайт отдаёт не обычную HTML-форму: autocomplete работает только через POST и
ограничивает ответ 15 группами, а расписание приходит в Drupal AJAX JSON.
Адаптер обходит ограничение autocomplete, повторяет временно повреждённые ответы
и публикует данные группы в БД транзакционно.

ИГЭУ использует ASP.NET WebForms. Его адаптер обходит все опубликованные виды
расписания, факультеты, курсы, группы и подгруппы, а экзамены, зачёты и сессии
сохраняет по точным календарным датам.
